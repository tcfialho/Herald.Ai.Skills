# Spike Execution (DEV Reference)

Execution contract for `SP-*` backlog stories generated by `/sm` from `SP-*` entries authored by `/po` in `.spec/spec.md ## Spikes`.

Authoring of `SP-*` entries (when to create a spike, mandatory triggers, the `spec.md` structure) lives in `../../po/references/spike_authoring.md`. `/dev` does **not** author spikes — DEV only executes them.

This document covers **where** spike artifacts live, **what** counts as a deliverable, **how** to propagate findings back into affected Use Cases, and **how** to close the spike.

---

## Artifact Layout

For a spike with ID `SP-NNN`:

```text
.spec/spikes/SP-NNN/
├── SP-NNN.md          # the research report — DEL-001 by convention
├── experiment.py      # or .sh, .ts, .ipynb, etc — DEL-002, DEL-003, ...
├── benchmark.txt      # or any other declared deliverable
└── ...                # every file listed under Deliverables (DEL-*) in the story
```

Rules:

- Create `.spec/spikes/` and the per-spike folder if missing.
- The research report path is `.spec/spikes/SP-NNN/SP-NNN.md`. Do **not** write spike outputs under repo-root `spikes/` or `docs/` — both are deprecated for spike artifacts.
- Every file under `.spec/spikes/SP-NNN/` **must** map to a declared `DEL-*` in the story. No undeclared files survive in this folder.

## Scratch vs. Deliverable

Any scratch artifact that is **not** a declared deliverable must live under `.temp/`, never under `.spec/spikes/SP-NNN/`. This includes:

- Throwaway probes and one-off API calls
- Raw API dumps, intermediate parsing output
- Debug logs and draft notes
- Downloaded fixtures used only during investigation

Scratch files must be deleted before the spike closes. The global temp-files rule applies.

If during the experiment a scratch file becomes load-bearing for the result, either:

1. Promote it to a `DEL-*` (update the story, then move the file to `.spec/spikes/SP-NNN/`), or
2. Capture the relevant content in the research report and delete the file.

A file under `.spec/spikes/SP-NNN/` that does not correspond to a `DEL-*` is an audit failure.

## Deliverable IDs (`DEL-*`)

Spike stories use `DEL-*` IDs instead of functional `AC-*`. Each deliverable has a stable ID (`DEL-001`, `DEL-002`, ...) declared by `/po` in `.spec/spec.md` and copied by `/sm` into the `SP-*` backlog story.

When completing a task that produces or finalizes a deliverable, use:

```text
spec task complete TASK-ID --agent <id> --covers DEL-001
```

This appends evidence to the story and synchronizes the `DEL-*` checkbox from the recorded `covers:` entry. Do not manually check `DEL-*` boxes without evidence.

## Decision Recommendation

Every spike report (`.spec/spikes/SP-NNN/SP-NNN.md`) must end with **exactly one** of the four declared Decision Options selected:

- **Discard** — no implementation; the spike answered "no" and there is nothing to build.
- **Implement** — create one or more `US-*` feature stories from the result. The new stories are produced in a follow-up `/sm` run, not inside the spike.
- **Create ADR** — record a durable architecture decision in `.spec/architecture.md`. The spike result locks a stack/integration/library choice.
- **Split into feature stories** — research is complete but delivery needs multiple `US-*`. Same follow-up `/sm` flow as Implement.

If the spike result demands a brand-new `US-*`, record it as a recommendation in the report. Do **not** silently fork new stories inside the spike. `/sm` creates them after the spike closes.

## Spike Use-Case Propagation (Mandatory)

Every `SP-*` story `/sm` produces carries a propagation contract. DEV is responsible for executing it before `submit-qa` closes the story.

### What the propagation task does

A propagation task (typically the last `TASK-*` in the story, named `TASK-XXX: Propagate spike findings to affected Use Cases`) edits the Use Cases whose AC, context, constraints, or wording change because of the spike result.

### Required structure

The `SP-*` story (authored by `/sm`) already provides:

1. **`## Affected Files`** — lists `.spec/spec.md` and every relevant `.spec/backlog/US-*.md` that derives from the affected UCs. If the story does not list these, return it to `/sm` — do not proceed.
2. **`write_scope`** — must cover all paths under `## Affected Files`. Without coverage, DEV cannot edit them. If a needed path is missing from `write_scope`, return to `/sm` to widen scope — do not edit outside scope silently.
3. **A dedicated propagation task** — `files:` lists the same UC paths, `verify_cmd` re-runs `python ../shared/scripts/spec.py docs validate`, and `covers:` references at least one `DEL-*` so evidence is recorded.
4. **A DoD line** — `- [ ] Affected Use Cases updated with spike findings (spec.md and every relevant US-*.md edited; paths listed under Affected Files).`

### DEV execution rules

- The propagation task **must** edit `.spec/spec.md` (Use Case sections impacted) and every relevant `.spec/backlog/US-*.md`.
- If the spike result invalidates an existing `US-*`'s AC or scope, the propagation task edits that `US-*` directly. Do not leave a stale story behind.
- If the spike result invalidates a story currently in flight (claimed by another agent or in QA), record the conflict in the spike report and pause — escalate to the user.
- Re-run `spec docs validate` as the verify command; the task is not complete until validation passes.
- A "no-change" outcome is rare and must be justified. If genuinely no UC is affected (pure infra spike with zero product surface change), the story should still list `.spec/spec.md` under `## Affected Files` with a comment explaining the no-change decision, and the report must elaborate so QA/audit can verify intent.

## Completion Gate

Before `spec story submit-qa` accepts the spike:

1. `.spec/spec.md` has `## Spikes` with the corresponding `SP-*` entry (authored earlier by `/po`).
2. Every `DEL-*` declared in the story has a matching file under `.spec/spikes/SP-NNN/` (or, when a deliverable is a document edit elsewhere, the path declared in the story).
3. The research report `.spec/spikes/SP-NNN/SP-NNN.md` exists, ends with one explicit Decision Option selected, and references each `DEL-*` it produced.
4. `.spec/spikes/SP-NNN/` contains **only** declared `DEL-*` artifacts. No scratch, no leftover debug logs, no intermediate dumps.
5. The propagation task has run, every UC path under `## Affected Files` was edited consistently with the spike result, and `spec docs validate` exits cleanly.
6. All tasks are complete with evidence. `covers:` IDs synchronize the `DEL-*` checkboxes.
7. `.temp/` no longer contains scratch produced by this spike.

When all of the above hold, run:

```text
spec story submit-qa --agent <id>
```

For `SP-*`, this command validates deliverables and evidence, then closes the story directly to `DONE` — spikes bypass `/qa`. After `DONE`, return to the Mandatory Loop in `../SKILL.md` step 2 and pull the next story.
